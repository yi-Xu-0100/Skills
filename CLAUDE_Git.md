# Git 操作规范

## 安全策略

### 允许使用的命令

以下命令无需确认，可直接执行：

| 命令 | 说明 |
|---|---|
| `git status` | 查看工作区状态 |
| `git diff` | 查看未暂存的差异 |
| `git log` | 查看提交历史 |
| `git branch` | 查看分支信息 |
| `git show` | 查看某次提交的详情 |
| `git stash` | 暂存当前改动（非破坏性操作） |

### 需确认的操作

| 命令 | 说明 |
|---|---|
| `git add` | 暂存文件。涉及大量文件时尤其要确认 |
| `git commit` | **执行前必须征得用户同意** |
| `git checkout` | 切换分支或恢复文件 |

### 严格禁止

以下操作禁止执行，除非用户**明确书面批准**：

- `git push` — 推送到远程仓库
- `git merge` — 合并分支
- `git rebase` — 变基操作
- `git reset --hard` — 强制重置（会丢弃工作区改动）
- `git clean -f` — 删除未跟踪文件
- `git checkout -- <file>` — 丢弃单个文件的改动
- `git branch -D` — 强制删除分支

### 安全原则

- 发现非本人造成的意外变更时，**立即停止**当前操作并告知用户，不得擅自处理。
- **永不**使用 `--no-verify`、`--no-gpg-sign` 等绕过 hook 的参数。
- **永不**对 `main`/`master` 分支执行 force push。
- **不 amend commit**，除非用户明确要求。提交失败时创建新提交而非 amend。

## 提交信息规范

### 格式

```
<type>(<scope>): <subject>

<body>
```

- 语言：中文
- 每次提交只做一件事，不同问题拆成多个小而聚焦的提交

### type 取值

| type | 用途 |
|---|---|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档变更 |
| `style` | 格式调整（不影响代码逻辑） |
| `refactor` | 重构（不增功能也不修 bug） |
| `test` | 测试相关 |
| `chore` | 构建、依赖、工具等杂项 |
| `perf` | 性能优化 |

### 主题规则

- 最多 **50 个字符**
- 使用**命令语气**（如 "添加" 而非 "添加了"）
- **不加句号**
- 说清变更的动机，而非罗列做了什么

### 正文规则（可选，复杂提交时使用）

- 说明 **"做了什么"** 和 **"为什么做"**
- 补充前置条件、影响范围、回滚方式等

### 示例

```
feat(lf-to-crlf): 新增 Git 报错场景的自动触发

当 git add/commit 遇到 "fatal: LF would be replaced by CRLF" 报错时，
自动提取文件路径并完成转换，无需手动扫描整个仓库。
```

```
fix(scripts): 修复 perl 在 PowerShell 下的路径转义问题
```

```
chore: 更新 .gitignore 忽略本地 Claude 配置文件
```

## 提交流程

1. 运行 `git status` 和 `git diff`，确认所有待提交的改动
2. 按上述规范编写提交信息，展示给用户确认
3. 用户同意后执行 `git commit`
4. 每次提交前有变更就运行项目的 lint/验证命令
